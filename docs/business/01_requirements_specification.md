# テニスコート予約システム 要件定義書

> **目的：** AIツール（Cursor、Google Antigravity等）を最大限に活用した開発を前提とした、精緻なビジネス・システム要件定義書

---

## 1. ビジネス要件整理

### 1.1 解決する課題（ペイン）

| 課題 | 詳細 |
|------|------|
| **予約管理の煩雑さ** | 電話やメールでの手動管理による、ダブルブッキングや転記ミスの発生 |
| **利用者側の利便性** | 空き状況がリアルタイムで分からず、予約の確定までに時間がかかる |
| **社内/組織内認証の壁** | 一般的なSNSログインではなく、既存のSalesforceアカウントでセキュアに管理したい |

### 1.2 ターゲット顧客（ペルソナとターゲット見込み数）

- **ペルソナ：** Salesforceを導入している企業の福利厚生担当者、またはテニスクラブ運営者
- **ターゲット数：** Salesforce利用企業のうち、福利厚生施設（テニスコート等）を自社保有または提携している企業

### 1.3 収益モデル

| パターン | 内容 |
|----------|------|
| **完全無料サービス** | すべて無料で提供。支払い情報の入力や管理は一切不要 |
| **管理コスト削減** | 予約管理の自動化による間接的な利益（ROI） |

### 1.4 競合との差別化

- **Salesforce連携：** 既存の従業員データや顧客データを活用した認証（SSO）により、新規ID発行の手間をゼロにする
- **特定ルール特化：** 「最大2時間」「土日祝限定」といった、汎用ツールでは設定が面倒な特定ルールをプリセット

### 1.5 効果的なアプローチ

- **LP一体型：** サービス説明、利用規約、ログインボタンが1枚で完結するシンプルな構成
- **Slack/Teams通知：** n8n連携により、予約完了時にチャットへ即時通知し、利便性を高める

### 1.6 MVP機能と正規版の切り分け

| フェーズ | 機能 |
|----------|------|
| **MVP** | メール/パスワード認証、予約カレンダー表示、予約実行・キャンセル（基本ルール遵守） |
| **正規版** | Salesforce SSO連携、n8n自動通知、コート複数対応、雨天中止の一斉通知、利用履歴のSalesforce側への書き戻し分析 |

## 2. 機能要件定義

### 2.1 MVP機能（完全無料版）

| 機能カテゴリ | 詳細 |
|--------------|------|
| **認証** | Supabase Auth（メール/パスワード）によるログイン・ログアウト、ゲスト予約対応 |
| **予約カレンダー** | 9:00-17:00（1時間単位）の空き状況表示。土日祝のみアクティブ |
| **予約登録** | 1ユーザー1日最大2時間まで。1時間単位で選択 |
| **予約確認** | 予約日時、会員情報、連絡事項、利用規約同意 |
| **予約完了** | 予約番号発行、確認メール送信 |
| **マイページ** | プロフィール確認、予約状況サマリー、最近の予約表示 |
| **プロフィール管理** | 会員情報の確認・編集（氏名、氏名カナ、メールアドレス） |
| **予約管理** | 予約履歴の表示、予約詳細の確認、予約の変更・キャンセル機能 |
| **バリデーション** | 同一時間帯の重複予約禁止。最大時間超過の制限 |
| **利用上の注意表示** | 来館申請の注意をトップ（ご利用ルール上）、予約カレンダー上、予約履歴・予約詳細の下部に表示。文言は `VISIT_APPLICATION_NOTICE`（`src/lib/constants.ts`）で一元管理 |

**注意：** 支払い関連機能（月額プラン、回数券、決済情報入力等）は一切実装しない。

### 2.2 正規版への拡張機能

- **Salesforce SSO：** Salesforceアカウントによるシングルサインオン認証
- **n8n自動通知：** 予約完了時・前日リマインドのメール/Slack通知
- **複数コート管理：** コートA、コートBの選択機能
- **雨天中止の一斉通知：** 天候による中止時の自動通知
- **管理者ダッシュボード：** 利用率のグラフ化、特定ユーザーの予約制限

## 3. 非機能要件

| 項目 | 要件 |
|------|------|
| **可用性** | VercelとSupabaseを利用し、99.9%以上の稼働率を確保 |
| **セキュリティ** | Supabase Authによる認証（MVP: メール/パスワード、正規版: Salesforce SSO追加）。データベース（Supabase）の行レベルセキュリティ（RLS）で他人の予約改ざんを防止 |
| **レスポンシブ** | スマートフォン（コート現場での操作）とPC（自宅での予約）の両方に対応 |

| レイヤー | 技術 | フェーズ |

### 4.2 技術スタック選定理由

| 技術 | 選定理由 |
|------|----------|
| **React/Next.js** | AIが最も得意とするコード生成対象であり、バグが少ない |
| **Supabase** | サーバー構築が不要で、SQLの生成もAIと相性が良い |
| **Vercel** | GitHub連携による自動デプロイが非エンジニアでも容易 |

### 4.3 データベース設計 (Supabase/PostgreSQL)

#### Table: `profiles` (ユーザー情報)

| カラム名 | 型 | 説明 |
|----------|-----|------|
| `id` | uuid | Primary Key, references auth.users |
| `full_name` | text | ユーザー名 |
| `full_name_kana` | text | ユーザー名（カナ） |
| `email` | text | メールアドレス |
| `created_at` | timestamp | 作成日時 |
| `updated_at` | timestamp | 更新日時 |

#### Table: `reservations` (予約データ)

| カラム名 | 型 | 説明 |
|----------|-----|------|
| `id` | uuid | Primary Key |
| `user_id` | uuid | Foreign Key to profiles.id |
| `booking_date` | date | 予約日 |
| `start_time` | time | 開始時間 |
| `end_time` | time | 終了時間 |
| `contact_notes` | text | 連絡事項（任意） |
| `reservation_number` | text | 予約番号（一意、自動生成） |
| `created_at` | timestamp with time zone | 作成日時 |

#### Check Constraints（制約）

```sql
-- 営業時間内の予約のみ許可
CHECK (start_time >= '09:00' AND end_time <= '17:00')

-- 1予約あたり最大2時間まで
CHECK ((end_time - start_time) <= interval '2 hours')
```

## 7. リスクと対策

#### MVP

| リスク | 詳細 | 対策 |
|--------|------|------|
| **同時予約（競合）** | ほぼ同時に予約ボタンを押した場合 | SupabaseのDB制約（Unique Index）により、同一時間・同一コートの重複を物理的に不可能にする |

#### 正規版（追加リスク）

| リスク | 詳細 | 対策 |
|--------|------|------|
| **SSO設定の難易度** | Salesforceの「接続アプリケーション」設定には専門知識が必要 | Supabaseの公式ドキュメントに従い、Cursorにエラーログを投げて修正する |
| **n8n連携の安定性** | 外部サービス連携による障害リスク | エラーハンドリングとリトライ機構の実装 |
