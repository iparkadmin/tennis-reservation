## 1. 個人情報保護 🔒

### 💡 一言で言うと
**「銀行の窓口で、自分の口座情報しか見られないのと同じ仕組み」**

---

### 📋 実施した対策

#### A. 本人確認の徹底

**例え話: 銀行の窓口**
- 銀行で口座情報を見るには、本人確認が必要です
- 他人の口座情報は見られません
- 予約システムも同じです

**システムでの対応**:
```
ログインしたユーザー
 ↓
【本人確認】メールアドレス + パスワード
 ↓
自分の予約だけ見られる ✅
他人の予約は見られない ❌
```

**保護されている情報**:
- 📝 予約者の氏名
- 📧 メールアドレス
- 💬 連絡事項（メモ）

---

#### B. 必要最小限の情報公開

**例え話: レストランの予約状況**
- レストランのホームページで「18時は満席」と表示
- でも、「18時に予約している人の名前」は見えない
- 予約システムも同じです

**公開する情報**:
- ✅ 2026年1月25日 10:00-11:00 コートA **予約済み**
- ✅ 2026年1月25日 11:00-12:00 コートA **空き**

**公開しない情報**:
- ❌ 誰が予約したか
- ❌ 予約者の連絡先
- ❌ 予約の連絡事項

**効果**:
空き状況は確認できるが、個人情報は守られる

---

#### C. すべての操作を記録

**例え話: 防犯カメラと入退室記録**
- オフィスビルの入退室記録（誰が、いつ、何をしたか）
- 不正があった時に確認できる
- 予約システムも同じです

**記録される内容**:
| いつ | 誰が | 何をした | 詳細 |
|------|------|---------|------|
| 2026/1/20 10:05 | 山田太郎さん | 予約作成 | 1/25 10:00-11:00 コートA |
| 2026/1/20 15:30 | 山田太郎さん | 予約変更 | 時間を11:00-12:00に変更 |
| 2026/1/22 09:15 | 山田太郎さん | 予約キャンセル | - |

**効果**:
- ✅ 不正な操作があれば、すぐに発見できる
- ✅ 問題発生時に、原因を特定できる
- ✅ コンプライアンス要件を満たす

## 2. 認証情報管理 🔑

### 💡 一言で言うと
**「金庫の鍵を、誰でも見られる場所に置かない」**

---

### 📋 実施した対策

#### A. 秘密情報の適切な管理

**例え話: 会社の金庫**
```
❌ 悪い例:
   「金庫の暗証番号: 1234」と書いた紙を、
   オフィスの壁に貼る

✅ 良い例:
   暗証番号は社長だけが知っている金庫に保管
   必要な人にだけ口頭で伝える
```

**システムでの対応**:

**修正前（危険）**:
```
📄 マニュアル.docx
「データベースのパスワード: password123」
→ このファイルを誰かに送ったら、パスワードが漏れる！
```

**修正後（安全）**:
```
📄 マニュアル.docx
「パスワードは別途管理。担当者に確認してください」

🔒 暗号化された金庫（.env.example）
「password123」
→ 許可された開発者だけが見られる
```

**効果**:
- ✅ マニュアルを外部に送っても安全
- ✅ パスワードが漏洩するリスクがゼロ
- ✅ 必要な人だけがアクセスできる

---

#### B. 情報の公開範囲の制限

**例え話: 社内文書の分類**
```
📗 社外秘（公開可）
   - サービス紹介
   - 利用規約

📕 機密（社内のみ）
   - データベースのパスワード
   - システムの設定情報
```

**対応内容**:

| ファイル | 公開範囲 | 状態 |
|---------|---------|------|
| README.md（サービス紹介） | 📗 社外秘 | ✅ 安全 |
| データベース設計書 | 📗 社外秘 | ✅ 安全 |
| パスワード・APIキー | 📕 機密 | ✅ 別管理 |

## 3. コンプライアンス 📜

### 💡 一言で言うと
**「個人情報保護法など、法律のルールを守っています」**

---

### 📋 対応状況

#### A. 個人情報保護法への対応

**法律が求めること**:
1. 個人情報を安全に管理すること
2. 本人以外が勝手に見られないようにすること
3. 誰がいつアクセスしたか記録すること
4. 情報漏洩が起きないよう対策すること

**システムでの対応**:

| 法律の要求 | システムの対応 | 状態 |
|-----------|--------------|------|
| **安全管理措置** | 暗号化通信（HTTPS）、パスワード保護 | ✅ 対応済み |
| **アクセス制御** | 本人のみがデータにアクセス可能 | ✅ 対応済み |
| **記録の保持** | すべての操作を自動記録 | ✅ 対応済み |
| **情報漏洩対策** | 認証情報の適切な管理 | ✅ 対応済み |

---

#### B. 監査証跡（何があったか記録）

**例え話: 会計帳簿**
- 会社の経理では、すべての取引を記録します
- 監査の時に「いつ、誰が、何をしたか」を説明できます
- システムも同じです

**記録される情報**:
```
【予約作成】
日時: 2026年1月20日 10:05:32
ユーザー: 山田太郎（user-id: 12345）
操作: 予約作成
内容: 2026年1月25日 10:00-11:00 コートA

【予約変更】
日時: 2026年1月20日 15:30:45
ユーザー: 山田太郎（user-id: 12345）
操作: 予約変更
変更前: 10:00-11:00
変更後: 11:00-12:00
```

**効果**:
- ✅ 監査時に必要な記録がすべて揃っている
- ✅ トラブル発生時に原因を特定できる
- ✅ 法的証拠として使用可能

---

##### 監査証跡の確認方法（記録をどう見るか）

**だれが確認できるか**  
- **システム管理者**（Supabase プロジェクトにログインできる権限を持つ人）のみ。  
- 一般の利用者やアプリの画面からは参照できません（不正閲覧を防ぐため）。

**どこで確認するか**  
- **Supabase の管理画面（ダッシュボード）** の **SQL Editor** を使います。

**前提（事前に必要な作業）**  
- `audit_logs` テーブルは、**`doc/17_additional_security.sql` を SQL Editor で実行したとき**に作成されます。  
- **未実行の場合**、下記の SELECT を実行すると `relation "audit_logs" does not exist` というエラーになります。  
  → 先に **`doc/17_additional_security.sql`** を開き、SQL Editor で実行してください。実行後、以降の予約の作成・変更・削除が自動で記録されます。

**手順（監査時の確認用）**

1. Supabase にログインし、当該プロジェクト（tennis court reservation）を開く。
2. 左メニューから **「SQL Editor」** を選択。
3. 次の SQL を入力して **実行（Run）** する。
   ```sql
   SELECT id, user_id, action, table_name, record_id, old_data, new_data, created_at
   FROM audit_logs
   ORDER BY created_at DESC
   LIMIT 500;
   ```
   （`audit_logs` が存在しない場合は、上記「前提」のとおり `17_additional_security.sql` を先に実行してください。）
4. 一覧が表示されます。
   - **action**: `create`（作成） / `update`（変更） / `delete`（削除）
   - **created_at**: 操作日時
   - **old_data** / **new_data**: 変更前・変更後の内容（JSON）
5. 監査用に残す場合は、結果エリアの **「Download as CSV」**（または「Export」）で CSV を取得できます。

**期間で絞る例**

```sql
-- 例: 2026年1月の記録だけ見る
SELECT * FROM audit_logs
WHERE created_at >= '2026-01-01' AND created_at < '2026-02-01'
ORDER BY created_at DESC;
```

**補足**  
- 専用の「監査ログビューア」画面はまだありません。上記のとおり、**SQL Editor での確認と CSV エクスポート**が現状の方法です。  
- 監査人が直接 Supabase に触れない場合は、**管理者が上記で CSV を出力し、監査人に提供**する運用になります。
